> 范围：单食堂当前形态。**不汇报多食堂（campus）**，仅作"展望"一句带过。
> 主线（贯穿始终）：**这不是一个带动画的界面，而是一个能配置、能运行、能自动诊断瓶颈并在受控条件下验证方案的食堂仿真系统。**

## Slide 2 — 仿真目标与系统架构（45s）

- **顶部目标句（先讲，口语化，本页主线）**：

  > 用仿真回答：窗口数、到达率变化时，排队时间和窗口利用率会怎么变？

- **再上架构图**（从上到下，三人色块标注——分工通过配色体现，不作标题）：

  ```
  前端（宋嘉桐）  4 页：参数配置 / 仿真运行 / 数据分析 / 历史记录
                  3D 数字孪生(Three.js) + 2D fallback + ECharts
                  自动诊断面板 / 干预建议卡 / 受控对比面板
        │  REST API（朱思思）：10 个接口 · 稳定 JSON 契约
        ▼
  仿真引擎（朱思思）  离散事件核心 ── 排队子模块 + 就餐子模块
        │
        ▼

  配置与分析（贾文霞）：默认参数标定 · 统计指标设计 · 效果验证
  ```

- **关键句**：架构是"怎么回答这个问题"，不是模块清单；接口先行，三人按模块并行开发互不阻塞。
- **口播**："我们要回答这个问题——架构就是为回答它服务的。前后端只通过 10 个接口对话。"

## Slide 3 — 核心设计亮点（45s）

- **后端仿真核心（1 句带过）**：离散事件驱动（`arrival`/`service_end`/`eat_end` + 堆队列按时间序），最短队列分配、就近座位、正态采样带下限。
- **决策闭环（本页重点，系统最大亮点）**：系统不止仿真，跑完会**自动**——
  1. **诊断**：判拥堵等级（低/中/高）、点名瓶颈（窗口排队 / 座位周转）
  2. **建议**：排序干预方案，自动算出建议配置（如基线 6 窗口 → 建议 8）
  3. **受控验证**：一键用**同一随机 seed** 自动重跑基线与建议方案，输出三指标对比
- **3D 数字孪生**：后端状态实时镜像（楼层堆叠、排队、座位、在途学生），非装饰。
- **两个硬亮点（答辩必讲）**：
  - `total_arrived == total_served` —— 仿真自洽性
  - 受控对比**两次同一 seed** —— 把"参数效果"和"随机噪声"分开，结论可信
- **配图**：左=决策闭环四步小图，右=3D 场景标注图。代码最多一小段事件循环伪代码。

## Slide 4 — 现场 Demo：自动诊断 + 受控 A/B（90s · 主菜）

**演示的是系统自己做实验，不是人点功能。**

**现场脚本：**

1. 「参数配置」→ 提交一组**确实拥堵**的基线（高到达率 + 窗口偏少，座位别太紧）→「开始仿真」→「仿真运行」3D 主屏。
2. 实时讲 ~35s：高峰下队列在 3D 里持续涨、拥挤色带变深，旋转 + 下钻一层。
3. 点「结束仿真」（后端 `/simulation/finish` 快进结算）→ 自动到「数据分析」。
4. **指系统自动诊断**："高拥堵 / 瓶颈：窗口排队 / 建议：增开窗口"，建议方案卡显示 **窗口 6 → 8（系统自己算的，不是我们手挑）**。
5. 点「**运行建议方案**」按钮 → 系统**自动用同一 seed 重跑基线 + 建议方案**（~20-30s，按钮显示"运行中…"）。
6. **受控对比面板自动弹出**：平均等待 ↓、峰值排队 ↓、座位利用率变化，带改善配色，面板标注 `seed=…`。

**口播要点**："注意——B 方案是系统诊断后自己建议的；两次用同一随机输入，所以差异只来自窗口数，这才叫受控对比。"

**现场实时跑保险（答辩前必做，逐项打勾）：**

- [ ] 后端已 `python backend/app.py`，浏览器开好 `http://127.0.0.1:5001/`
- [ ] **基线参数标定为"窗口瓶颈"**：预演确认数据分析页诊断为中/高拥堵、主建议=「增开服务窗口」、「运行建议方案」按钮**可点**（不可点 = 没触发瓶颈，需调高到达率 / 减窗口；座位别设太紧，否则系统会先建议加座位）
- [ ] 预演确认两次自动重跑总耗时可接受（基线 `total_time` 别设太大；必要时缩短仿真时长加快两次 `finish`）
- [ ] 备 2-3 张截图 / 30s 录屏作 fallback（含受控对比面板那一刻）
- [ ] 兜底话术：系统保留 2D fallback 可一键切换——既是设计卖点也是演示保险
- [ ] 记得现场指着面板上的 `seed=…` 讲"同一随机输入"，这是评分点

- **配图**：3D 运行分镜 2 张 + 自动诊断卡 + 受控对比面板各 1 张

## Slide 5 — 成果与数据分析：回扣目标（45s）

- **核心是那块"受控对比面板"**（系统自动产出，不是手工叠加）：

  | 指标 | 基线 | 建议方案 | Δ |
  |------|------|----------|---|
  | 平均等待 | … | … | ↓（改善配色） |
  | 峰值排队 | … 人 | … 人 | ↓ |
  | 座位利用率 | …% | …% | 变化 |

- **旁证**（各一句，不展开）：自动诊断卡（等级/瓶颈/建议）、干预建议卡、窗口分布与队列时间线 ECharts。
- **结论句（回扣第 2 页目标）**：

  > 系统能自动诊断瓶颈，并在同一随机条件下量化配置调整带来的收益。

- **口播**："回到开头那个问题——系统自己判断窗口是瓶颈，自己建议加到 8，并在受控条件下证明等待时间真的降了。"
- ⚠️ **口径准确**：后端输出的是**座位利用率** + 窗口服务分布，没有单一"窗口利用率"字段；答辩别说成"窗口占用率"。受控对比的具体数值 → 贾文霞预演时记录真实输出填表。
