## 1. 本阶段承担的任务

负责"北京交通大学就餐仿真系统"的全部后端开发与团队节奏把控，具体包括：

1. **仿真引擎核心**：基于离散事件驱动模型的 `SimulationEngine`，事件包括 `arrival` / `service_end` / `eat_end` 三类，使用堆队列保证按时间序处理。
2. **排队子模块**：`Window` 类与最短队列分配策略，打饭时长采用正态分布采样并加下限保护。
3. **就餐子模块**：`Seat` 类、就近座位分配（按 `window_id × seat_per_window` 映射）、就餐时长正态采样且下限 60 秒。
4. **REST API**：9 个接口（参数配置、启动、单步、状态、暂停、结束、重置、统计、历史），使用 Flask Blueprint 组织。
5. **数据持久化**：SQLite 两张表（`simulation_config`、`simulation_snapshot`），快照采用 50 条批量缓冲落库以减少 I/O。
6. **单元测试**：pytest 38 条，覆盖三层模块 + API，0.39 秒跑完。
7. **小组协调**：组织 4 次小组沟通，统一接口约定与开发节奏。

## 3. 个人遇到的问题与解决过程

### 3.1 macOS 5000 端口被 AirPlay 占用

**现象**：`python app.py` 启动后报 `Address already in use`。
**排查**：通过 `lsof -ti :5000` 确认是 AirPlay Receiver。
**解决**：把 Flask 默认端口改成 5001，README 中也注明这一点，避免队友重复踩坑。

### 3.2 `total_arrived` 一上来就是最终人数

**现象**：仿真刚跑了 0.43s，前端就显示已到达 98 人；正在跑的实例里也出现了 `current_time=746.10s` 时 `total_arrived=307`、`total_served=1` 的反直觉数据。
**根因**：`_generate_arrival_events()` 在 `start()` 时一次性预生成所有到达事件，而我把 `self.total_arrived += 1` 写在了循环里，导致总人数在仿真开始前就被填满。
**解决**：把累加移到 `_handle_arrival()` 事件处理函数里，让 `total_arrived` 真正表示"截至当前时刻已到达"。补回归用例 `test_total_arrived_zero_right_after_start`。
**反思**：把"事件预生成"和"事件处理"语义混在一起是设计上的味道，下次类似场景应该明确区分"事件就绪"和"事件发生"两个阶段。

### 3.3 `/simulation/start` 可重复调用污染事件队列

**现象**：连续两次提交配置后，事件队列长度直接翻倍。前端表单防抖不够时容易出现。
**根因**：`engine.start()` 没有幂等守卫，每次调用都会再跑一次 `_generate_arrival_events()`。
**解决**：在 `SimulationEngine.__init__` 加 `self._is_started = False`，`start()` 进入时判断；路由层也对应返回 `already_started=True` 提示。
**反思**：所有"初始化型"方法都应该考虑幂等性，这次教训记下来。

### 3.4 "结束仿真" 按钮其实是暂停

**现象**：老师点"结束仿真"看到的是半截统计，不是最终结果。
**根因**：前端按钮调的是 `/api/simulation/pause`，没有"快进到结束"的语义。
**解决**：新增 `/api/simulation/finish`，后端在循环里 step 直到 `is_ended`，分段刷库避免内存爆。前端按钮调用此接口，结算期间显示"结算中…"并禁用按钮。
**反思**：交付物里出现的语义偏差是最致命的，这种 demo 级 bug 老师一点就破。后面写代码会更敏感。

### 3.5 座位热力图的"剩余时间"是假的

**现象**：座位的 `remaining_time` 入座时设一次值，离座时清零，中间从不递减；前端拿它做颜色深浅却"永远不变"。
**根因**：把"剩余时长"当成了静态字段。
**解决**：`Seat` 改存 `eat_end_time`（绝对时刻），`_build_state()` 渲染时按 `current_time` 动态换算 `remaining_time = max(0, eat_end_time - current_time)`。补回归用例 `test_remaining_time_decreases_for_same_seat`。
**反思**：状态字段应该尽量保留"原始信息"（绝对时刻），派生量按需计算，不要存"快照时刻的差值"。

### 3.6 历史记录前端缺入口

**现象**：README 写了"SQLite 快照记录，可查历史"，但前端只有 3 页，没有历史入口。
**解决**：后端补 `/api/history/configs` 给配置列表 + 汇总；前端加第 4 个 nav 与历史页（表格 + 点击展开 ECharts 多线图）。

### 3.7 学生流动动画

**现象**：后端算了 `data.students` payload，前端却只画聚合圆点，根本不是"人员流动"。
**解决**：`drawStudentDots()` 用 `student.id` 追踪 + 帧间线性插值（lerp=0.4）。窗口排队、正在打饭、等位三类都按学生 ID 渲染独立圆点，从一个位置滑动到下一个位置。

## 5. 阶段性反思

1. **接口先行的好处**：后端在第一周就把 9 个接口的字段 lock 死，前端拿着固定 schema 独立开发，整个阶段没出现过"等接口"的卡点。
2. **测试发现的 bug 比设计阶段多**：5 个核心 bug 全是测试 + 自查时发现的，靠 review 文档发现不了。这个体验让我以后会更早写测试。
3. **小组节奏靠的是固定例会**：每周三晚 20:00 雷打不动，议题清单提前一天发到群里，比临时拉会效率高。
4. **AI 不能替代理解**：AI 生成的代码我每行都读过、跑过测试，几次老师抽查我都能讲清逻辑。这一点不能侥幸。
