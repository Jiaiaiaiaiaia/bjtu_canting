## 2. 划分原则

按"模块 + 文件归属"做硬切分，模块之间通过 REST API 这一稳定接口交互，确保三人可以并行开发、互不阻塞：

- 后端只关心 JSON 输入输出，不关心前端如何渲染；
- 前端只调 `/api/*` 接口，不关心仿真引擎如何实现；
- 配置与分析模块对接的是已有 API 与文档，不直接动代码主体。

这种方式的好处是单元测试可以独立进行，集成阶段只需要做接口联调。

## 4. 接口约定（已稳定）

后端为前端暴露 9 个 REST 接口，均返回 JSON：

| Method | Path | 用途 |
|--------|------|------|
| POST | `/api/config` | 提交参数配置，初始化引擎 |
| POST | `/api/simulation/start` | 启动仿真（带防重启动守卫） |
| GET  | `/api/simulation/step` | 单步推进，返回当前状态快照 |
| GET  | `/api/simulation/status` | 查询当前是否正在运行 |
| POST | `/api/simulation/pause` | 暂停轮询 |
| POST | `/api/simulation/finish` | 一次性跑完剩余事件，返回最终统计 |
| POST | `/api/simulation/reset` | 重置引擎 |
| GET  | `/api/statistics` | 获取当前统计 |
| GET  | `/api/history/configs` | 历史配置列表（含汇总） |
| GET  | `/api/history?config_id=N` | 指定配置的快照时间序列 |
